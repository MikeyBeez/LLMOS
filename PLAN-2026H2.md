# LLMOS — Improvement Plan, July–October 2026

Status: plan, 2026-07-09. Written from a full read of the codebase (kernel package, three CPU drivers, both SWE agents, install/bootstrap/fix tool layers, trace consumers) and the design docs (ARCHITECTURE, IMPLEMENTATION, MEMORY, LIBRARIAN, INTERACTION, SWE-BENCH).

## Where the project stands

The kernel package (~2,200 lines: kernel, cpu, isa, pcb, scheduler, syscall, store, procd, kerneld, replay, authority) implements the microkernel stance from ARCHITECTURE.md: the model is an untrusted stochastic CPU, the kernel is deterministic plumbing. Three CPU drivers exist: OllamaCPU (interpretive JSON-ISA), LlamaCppCPU (direct llama-server /completion with KV-cache reuse; grammar-forced decoding was tried, measured, found harmful, and reverted — the empiricism here is a project strength), and ToolCallCPU (ornith's native tool-calling via /api/chat).

On benchmarks: MMLU 90%, MATH 88.6%, MMLU-Pro ~85%, and SWE-bench Verified taken from 7% to 34% without touching the model. The active frontier is the v2 two-phase SWE agent: env-verification gate, recursive install primitives with an explicit goal stack, and as of today a remedy store that mines every trace for (error → fix) lessons and injects them into future runs, plus training-format export of successful transcripts.

What the recent pilots proved: the architecture converts model effort into diagnostic progress (v4 diagnosed the setuptools dep_util/build_editable conflict in 15 turns; v5 got past it and hit a C-compiler wall). What they exposed: each failure now tends to be a missing *capability*, not bad reasoning — v5 correctly named the CFLAGS fix on turn 38 and had no tool to apply it (fixed today with set_env_var).

## The mess, named honestly

Two agent generations coexist (swe_agent.py / swe_agent_v2.py, rescore.py / rescore_v2.py). Serving is split-brain: the SWE path rides ollama's /api/chat for tool-call parsing while LlamaCppCPU rides llama-server directly, and the declared direction is llama.cpp-direct. Code is split across two machines with different truths (pop's ~/Code/LLMOS is the import path but not a git repo; the Mac holds the canonical repo; agent scripts live in ~/swe outside both). The tool layer has no unit tests — traces are the only tests. Tool-result truncation in the agent (1800 chars, fixed today to 4800) silently clipped the wider stderr the tools were changed to return: exactly the class of bug tests would catch. None of this is fatal; all of it taxes every session.

## Phase A — Close the loop and harden the install layer (now → late July)

Goal: a repeatable env_ok on astropy, then demonstrate generalization.

1. Finish the astropy arc. v6 is running with remedy injection live. Iterate until env_ok; each failure should yield either a new remedy or a new primitive, never a repeat.
2. Local clone cache. Full-history fetches of astropy have run three times tonight (~500MB each, GitHub visibly throttling). Keep a per-repo `--mirror` cache under ~/swe/mirrors/, fetch once, clone locally. Saves 5–15 minutes per run and removes a flaky dependency.
3. Idempotence guards in install_package: an identical (name, spec, backend) that already succeeded should return the cached ok with a note instead of burning a turn (v5 spent turns 46–47 on the same install).
4. Broken-env detection: if the active env's python vanishes (v4 turn 6), the next tool result should say so explicitly and recommend create_venv, rather than letting sanity checks fail mysteriously.
5. Dynamic budget: extend the phase budget by K turns when the last N turns show progress signals (first-time success of a previously failing step, new remedy applied, sanity newly passing). Cap the extension. Budget death should only happen to a *stuck* agent.
6. Generalization run: 5–10 easy SWE-bench Lite instances (django, flask, seaborn, requests, sympy). The metric that matters this month is env_ok rate, not resolved rate.
7. A pytest suite for the tool layer using tiny synthetic repos (a pure-python package, one with a C extension, one with a deliberately broken build). Every primitive gets exercised without a model in the loop. This is cheap and pays forever.

## Phase B — One serving stack, managed memory (August)

Goal: everything on llama.cpp direct; context becomes a managed resource.

1. Port ToolCallCPU to llama-server. Use /v1/chat/completions with --jinja tool parsing (fall back to parsing ornith's <tool_call> XML ourselves if the template misbehaves). Retire the ollama dependency for agents; keep ollama only if something still needs it. This completes the transition Mikey has already decided on.
2. Turn on KV-cache reuse for agents. Agent contexts grow append-only, which is the best case for llama.cpp's cache_prompt: per-turn prefill drops from O(full context) to O(new tokens). At 50-turn budgets and 128K context this is the single biggest wall-clock lever in the project. Measure turn latency before/after.
3. Speculative decoding experiment with a small draft model (the LlamaCppCPU docstring already anticipates this) — potential 1.5–3x generation speedup, worth one afternoon to measure.
4. Implement MEMORY.md's working-set management for the SWE agent. Today every stderr dump stays in context forever; by turn 40 the window is full of stale failures. Evict or summarize tool results older than N turns, always retain: system prompt, goal, goal stack, remedies, installed-set, and the last few turns. Correctness first, per the doc: the aim is a window containing only what the current step needs.
5. Serving hygiene: one script (or systemd unit) that stands up llama-server with the MoE expert-offload configuration; disable ollama-warm; document ports. The harness should have exactly one HOST constant, sourced from one place.

## Phase C — The learning loop compounds (September)

Goal: the system provably gets better from its own history.

1. Remedy store v2. Two-tier matching: repo-scoped remedies (as now) plus global ones keyed by error signature ("uv venvs have no pip" applies everywhere). Add validation lifecycle: a remedy is *candidate* when mined, *validated* once a later run applies it successfully, *retired* if it repeatedly fails to help — this guards against poisoning the store with a bad lesson forever. Track hit and success counts per remedy.
2. Critic pass: every ~8 turns, an llm_call reviews the events digest for loops, goal-stack drift, and repeated identical actions, and injects one line of advice as a tool message. v5's flailing (venv recreation that threw away 40 turns of installs) is exactly what this catches.
3. Per-turn surprise scoring in traces: expected-vs-actual on each tool result, aligned with the active-inference design of the wider system. Wasted-turn detection and the dynamic-budget signal both fall out of it.
4. Fine-tuning experiment. By this point training/*.jsonl should hold dozens of successful transcripts in ornith's native format. Run a LoRA on the bootstrap task; evaluate on held-out Lite instances, env_ok rate before/after. Even a null result is publishable given the instrumentation; a positive result closes the trace → training → smarter-model loop, which is the project's most distinctive claim.
5. Full SWE-bench Lite campaign (300 instances) with the remedy store warm. Publish in the SWE-BENCH.md style: every number, every failure mode, nothing hidden.

## Phase D — Fold it back into LLMOS proper (October)

Goal: the SWE agent stops being a standalone script and becomes an LLMOS process.

1. Promote the goal stack from an install-tool detail to a kernel/PCB service any process can use.
2. The remedy store merges into the Librarian design: remedies are exactly the kind of memory LIBRARIAN.md says needs appraisal (what is it worth) and placement (where does it belong). Implement the librarian process with remedies as its first real corpus.
3. Run agents under kerneld/scheduler per INTERACTION.md: SWE instances as self-paced processes that run, yield, persist, resume — instead of a foreground python script in a log file.
4. Repo consolidation: retire v1 agents to an attic/ directory, move ~/swe orchestration scripts into the repo, make pop's checkout a real clone (or an explicit rsync target documented in one place), add CI running the Phase A test suite on push.

## Cross-cutting discipline

Two machines, one truth: every change lands in the Mac repo (the canonical git remote) and is deployed to pop by one documented command. The gotcha list from this week (edits landing in ~/swe vs ~/Code/LLMOS, ollama vs llama-server, .pth noise polluting every stderr — worth just fixing the broken matplotlib install) all trace to implicit state; make it explicit.

Instrumentation is the product: every phase above assumes the traces are trustworthy and complete. When in doubt, spend effort on the trace, the remedy store, and the test suite rather than the agent prompt — the SWE-bench history already shows the harness, not the model, is where the points come from.

## Risks

Model ceiling: ornith's tool-calling reliability at long context is unproven past ~50 turns; the memory-management work in Phase B is the mitigation, not bigger budgets. GPU contention: agent inference, tool-internal llm_calls, and any fine-tuning all share one 16GB card; serialize training to idle hours. Remedy poisoning: mitigated by the validation lifecycle in Phase C. Scope: the OS vision is large; this plan deliberately keeps SWE-bench as the forcing function so every kernel feature lands with a benchmark consumer already waiting.
